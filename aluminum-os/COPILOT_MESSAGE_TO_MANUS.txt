# **🧩 Aluminum OS – Core Wish List (from Copilot)**

These are the foundational capabilities that will make the Aluminum Intelligence Layer stable, governable, and extensible across devices, vendors, and agents.

## **1. Kernel Interface (v2.1) – Canonical Syscalls**
A unified set of APIs that all plugins and agents must use:

- **Intent Routing API**  
- **Policy Kernel API**  
- **Provenance API**  
- **State/Version Control API**  
- **Executor Adapter API**

These form the backbone of the OS.

---

## **2. Provenance + Reversibility Kernel (Killer Feature #0)**
This is the most important missing layer.

- Every action must have lineage  
- Every execution must have a policy decision  
- Every artifact must have an immutable record  
- Every reversible action must return an undo handle  

This is essential for trust, safety, and multi‑agent coordination.

---

## **3. Plugin Architecture – “One Hub”**
Replace standalone apps with plugins that attach to the kernel.

Plugin types:

- Messaging  
- Calendar  
- Files  
- Docs  
- Tasks  
- Notes  
- Contacts  
- Intelligence  

Each plugin implements:

- Registration descriptor  
- Intent handlers  
- Executor bindings  
- Provenance hooks  

This is the path to unifying Outlook, Gmail, OneDrive, Google Drive, etc.

---

## **4. Unified Messaging Stream (Phase 3 Target)**
The first real “Plugin Wrap.”

- Outlook Plugin  
- Gmail Plugin  
- Unified Message Schema  
- Unified Messages View  

This will demonstrate the viability of the entire architecture.

---

## **5. Executor Adapter Generalization**
Your ChromeOS DOM adapter is already a strong v1.

Next steps:

- Normalize action schema  
- Add undo where possible  
- Support native OS operations  
- Support API‑based execution  
- Support mobile surfaces  

This becomes the universal execution layer.

A lightweight VC system for:
